# Fernet Compatibility Test — RESULT

**Run date:** 2026-04-12
**Purpose:** Decide whether DFARS Desktop v2 (Rust) can read secrets encrypted by v1 (Python) without a migration step.

## Verdict

**COMPATIBLE** — Python `cryptography` 45.0.7 `Fernet` tokens round-trip to/from Rust `fernet` 0.2.2 byte-for-byte in both directions.

## What this means for the v2 rebuild

- v1's Fernet-encrypted secrets (TOTP secrets, Agent Zero API keys) will decrypt unchanged in v2.
- **No AES-GCM rewrap migration is needed.**
- The same Windows Credential Manager key entry (`service="dfars_desktop", account="fernet_key"`) is used by both apps.
- Spec §7 "Fernet compatibility" → confirm path taken; §13 Q1 → resolved.

- Rust `fernet` crate version: **0.2.2**, with `rustcrypto` feature (pure-Rust AES-CBC + HMAC-SHA256, no OpenSSL → MSVC-safe)
- Python `cryptography` version: **45.0.7**

## Why it worked

Both libraries are faithful implementations of the [Fernet spec](https://github.com/fernet/spec/blob/master/Spec.md):

- AES-128-CBC + PKCS#7 padding
- HMAC-SHA256 for authentication
- Timestamp prefix + IV + ciphertext + MAC
- URL-safe base64 encoding throughout
- Keys are 32 bytes URL-safe-base64-encoded (16 bytes signing + 16 bytes encryption, concatenated)

The key format is identical across implementations — a key generated by either library is directly usable by the other.

None. This is a green-light for keeping Fernet in v2. Spec §7 should be updated to remove the "verify before phase 1" caveat and the AES-GCM fallback plan.
