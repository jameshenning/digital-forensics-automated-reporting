# SEC-1: Auth Architecture Review
**DFARS Desktop v2 — Pre-Implementation Security Gate**
**Reviewer:** security-compliance-auditor
**Date:** 2026-04-12
**Status:** APPROVED WITH CONDITIONS
**Phase 1 implementation:** BLOCKED pending resolution of MUST-DO items 1 and 2 (keyring name mismatch + lockout timer hardening). All other work may proceed in parallel.

---

## 1. Executive Summary

The v1 auth architecture is well-constructed for a single-user forensics desktop: Argon2id everywhere, encrypted TOTP secrets, single-use recovery codes, and Fernet-encrypted secrets at rest are all appropriate choices. The v2 design inherits these strengths and the proposed session management change (in-memory HMAC map replacing the persisted session secret) is a genuine improvement in the Tauri context, not a regression.

However, two issues must be resolved before Phase 1 code can be written. First, the v2 spec names the wrong Credential Manager service/account keys — it says `service="dfars_desktop", account="fernet_key"` but v1 actually uses `service="DFARS Desktop", account="totp_encryption_key"`. Implementing against the spec names would silently create a new keyring entry on first v2 launch, leaving the existing TOTP secret and Agent Zero API key unreadable. Second, v1's lockout timer uses `datetime.now()` (wall-clock) stored as an ISO string in SQLite; if the user winds the system clock backward by 5+ minutes the lockout is bypassed. v2 must use a monotonic counter approach.

There are six additional SHOULD-DO items covering recovery code brute-force hardening, the Agent Zero auto-setup path's hardcoded filesystem path, the missing TOTP lockout (separate from the password lockout), the recovery code entropy quantification, the file-fallback key path security, and custody command auth-guard verification. No findings here rise to CRITICAL for a single-user, local-only app, but the keyring mismatch is operationally equivalent to data loss for the existing user.

## 2. Findings by Area

### 2.1 Password Hashing — Argon2id Parameter Interop

**v1 behavior:** `argon2-cffi`'s `PasswordHasher()` default constructor uses `time_cost=3, memory_cost=65536 (64 MB), parallelism=4, hash_len=32, salt_len=16`, algorithm Argon2id. Hashes are stored as the standard PHC string format: `$argon2id$v=19$m=65536,t=3,p=4$<salt>$<hash>`.

**v2 proposal:** `argon2` 0.5 (RustCrypto) with `Argon2::default()`. The RustCrypto `argon2` crate's `Default` impl uses `time_cost=2, memory_cost=19456 (19 MB), parallelism=1`, which does **not** match v1's parameters.

**Risk:** The defaults differ, but this is not a compatibility break. `Argon2::default().verify_password(&hash_bytes, &PasswordHash::new(&stored_hash))` reads the parameters from the PHC string header in `stored_hash` and verifies against those, not against the current default. This is how PHC-format hashes work — the parameters are self-describing. Verification will succeed for all v1 hashes. Future hashes generated by v2 (e.g., on password change) will use v2's weaker defaults unless explicitly overridden.

**Verdict:** PASS with a hardening requirement. Verification cross-compat is confirmed by the PHC self-description property. However, the v2 `Argon2::new()` call for *generating* new hashes (password set, password change, recovery code hashing) must use explicit parameters matching v1 rather than the Rust crate defaults, to preserve equivalent work-factor for new credentials.

**Implementation requirement:** When constructing `Argon2` for *hashing* (not verifying), use explicit parameters: `Argon2::new(Algorithm::Argon2id, Version::V0x13, Params::new(65536, 3, 4, None).unwrap())`. For *verification only*, `Argon2::default()` is acceptable since params come from the stored hash. Add a `#[cfg(test)]` round-trip test that generates a hash with these params, verifies it, and also verifies a known v1-generated hash string.

### 2.2 TOTP MFA — Parameter Interop with pyotp

**v1 behavior:** `pyotp.TOTP(secret)` defaults: SHA-1, 6 digits, 30-second step. `pyotp.random_base32()` generates a 32-character Base32 secret (160 bits). Verification uses `valid_window=1` (accepts the previous and next 30-second window, so ±60 s clock drift tolerance).

**v2 proposal:** `totp-rs` with `Algorithm::SHA1`, `digits=6`, `step=30`, same Base32 secrets.

**Risk analysis:**
- SHA-1 in TOTP (HOTP/TOTP RFC 6238) is not a cryptographic weakness — the HMAC-SHA1 construction is still sound for this use. SHA-256/SHA-512 TOTP exists but requires re-enrollment and is not supported by all authenticator apps. Matching pyotp is correct.
- `totp-rs` must be initialized with `step=30`, `digits=6`, `skew=1` (to match `valid_window=1`). The `skew` parameter in `totp-rs` is equivalent to `valid_window` in pyotp — it allows ±1 window.
- Base32 secrets from pyotp are stored encrypted in the DB. After Fernet decrypt, the plaintext Base32 string is passed to `totp-rs`'s `TOTP::new()` or `TOTP::new_unchecked()`. Verify the crate accepts raw Base32 strings (not byte arrays) at construction. If it requires decoded bytes, apply `base32::decode(Alphabet::RFC4648 { padding: false }, &secret_str)`.

**Verdict:** PASS. Parameters are compatible. Clock drift handling requires explicit `skew=1` in the `totp-rs` constructor, which must not be left at default.

**Implementation requirement:** Construct `TOTP` with `skew: 1` explicitly set. Add a test that generates a code from a known secret+timestamp in Rust and verifies it against a known pyotp output for the same inputs.

### 2.3 Recovery Codes — Format, Entropy, and Cross-Version Read

**v1 behavior:** `_format_recovery_code()` generates codes as two 5-character hex groups: `xxxxx-xxxxx`. The generation is: `secrets.token_hex(3)[:5]` for each group (3 bytes = 6 hex chars, truncated to 5). This gives **5 bits per character × 10 characters = not quite right**. Let's be precise: hex has 4 bits/character, so 5 hex chars = 20 bits per group, 10 hex chars = 40 bits total. With the hyphen separator and 10 codes, each code has ~40 bits of entropy. This is below the NIST SP 800-63B recommendation of 112 bits for out-of-band authenticators but is acceptable for a rate-limited single-user local app. The codes are Argon2id-hashed before storage.

**Recovery code verification (`verify_recovery_code`):** Fetches all unused code hashes for the user and iterates, calling `_hasher.verify()` for each. With 10 codes and ~100ms per Argon2 verify, worst-case is ~1 second to exhaust all codes. There is **no rate limiting on recovery code attempts** — a local attacker with a logged-in WebView could hammer `auth_verify_mfa` with recovery code attempts at ~1/sec without lockout.

**v2 compat:** Schema unchanged. Argon2id hash format self-describing. v2 reads and verifies them identically. PASS for cross-version compat.

**Verdict:** PASS for compatibility. CONCERN for missing brute-force throttle on recovery code verification (see SHOULD-DO #4).

### 2.4 Account Lockout — Wall-Clock Time Vulnerability

**v1 behavior:** On 5th failed attempt, `_record_failed_attempt()` computes `locked_until = datetime.now() + timedelta(minutes=5)` and stores it as an ISO string in SQLite. The check in `verify_password()` reads `locked_until_raw` from the DB, parses it with `datetime.fromisoformat()`, and compares to `datetime.now()`.

**The vulnerability:** `datetime.now()` returns the system wall clock. If the user (the local attacker in this threat model) winds the Windows system clock backward by 5+ minutes while locked out, `locked_until > datetime.now()` evaluates to False and the lockout is immediately bypassed on the next login attempt. The counter `failed_login_count` is reset to 0 on successful login, so after the clock bypass succeeds they can re-lock and repeat.

**Practical impact in this threat model:** The single user is also the owner, so a malicious local attacker capable of changing the system clock is arguably already privileged. However, this matters in scenarios where a forensic machine is seized and an adversary attempts offline or console access: the investigator's account could be bypassed, allowing evidentiary record manipulation. For a chain-of-custody app, this gap should be closed.

**v2 mitigation path:** Tauri commands run in a persistent Rust process. The lockout state can be held in a `Mutex<HashMap<String, Instant>>` in `AppState` as an in-memory monotonic counter. The approach: store `failed_count` and `lockout_expires: Option<std::time::Instant>` in `AppState.auth_state`, checked on every login attempt. Persist the count to SQLite for cross-restart durability, but derive lockout expiry as `Instant::now() + duration` at process startup if `failed_login_count >= MAX_FAILED_ATTEMPTS && locked_until IS NOT NULL` in the DB. `Instant` is monotonic and immune to system clock changes.

**Verdict:** FLAG — must fix in v2. This is the second MUST-DO item.

### 2.5 Session Management — Proposed Design Change

**v1 behavior:** Flask session cookie signed with `SECRET_KEY`, which is a 32-byte random hex string persisted in `config.json`. Sessions are permanent (7-day lifetime). Cookie flags: `HttpOnly=True`, `SameSite=Lax`, `Secure=False` (acceptable since it only serves 127.0.0.1). The session dict holds `username` (fully logged in) or `mfa_pending` (password passed, awaiting MFA).

**v2 proposal:** Drop the persisted session secret entirely. Use an in-memory session map in `AppState`: `Mutex<HashMap<SessionToken, SessionData>>`. On login, generate a random token, store it in the map, and return it to the React frontend. Subsequent Tauri commands receive the token and look it up. Because Tauri commands authenticate the active WebView process (not an arbitrary HTTP client), there is no cross-origin session theft risk.

**Analysis:**

This is a genuine security improvement for the Tauri context, for the following reasons:

1. **No persisted secret to steal.** v1's `config.json` contains the session secret in plaintext. Whoever reads that file can forge session cookies. v2 has no equivalent persistent secret.

2. **Sessions die with the process.** In-memory sessions expire on app restart. This is appropriate for a desktop forensics app where the investigator should re-authenticate after a crash or update.

3. **No cookie attack surface.** Tauri commands do not go over HTTP. The WebView cannot set cookies that affect Tauri IPC. The session token is passed via Tauri's IPC serialization, not an HTTP cookie header — so there is no `SameSite`, CSRF, or cookie-theft concern.

4. **No secret management overhead.** The v1 session secret in `config.json` is a flat, unencrypted 32-byte hex string accessible to any process that can read `%APPDATA%\DFARS\config.json`. The v2 in-memory map has no on-disk representation.

**Risks to verify in implementation:**

- The session token must be generated with `rand::thread_rng().gen::<[u8; 32]>()` (or equivalent cryptographically secure RNG), not a predictable value.
- Session lookup must be O(1) (HashMap) and the token comparison must be constant-time (`subtle::ConstantTimeEq` or equivalent) to prevent timing-based session oracle attacks from other processes on the same machine.
- The session map must distinguish between `Pending(username)` (password verified, MFA not yet done) and `Active(username)` states — matching v1's `mfa_pending` vs `username` session keys.
- `auth_logout()` must remove the token from the map immediately (no TTL-only expiry).
- The spec says the HMAC cookie is "issued to the WebView on login." Clarify in implementation: this should be a random opaque token passed as an IPC return value, not an actual HTTP cookie. If Tauri's WebView has any mechanism to exfiltrate cookies to JavaScript (e.g., via `document.cookie`), avoid the cookie model entirely in favor of a token stored in React component state or `sessionStorage` (which is cleared on WebView close).

**Verdict:** APPROVED. The design is sound. Implementation must follow the specifics above.

### 2.6 Fernet Key in Credential Manager — Service/Account Name Mismatch

**v1 actual behavior (`crypto.py`):**
```
_KEYRING_SERVICE = "DFARS Desktop"
_KEYRING_USERNAME = "totp_encryption_key"
```

**v2 spec §7 states:**
> v1 uses `keyring.set_password("dfars_desktop", "fernet_key", ...)`. Rust `keyring` crate hits the same Credential Manager entry.

**These are wrong.** The spec has both the service name and the account name incorrect:
- Service in v1: `"DFARS Desktop"` (with space and capital D, not `"dfars_desktop"`)
- Account in v1: `"totp_encryption_key"` (not `"fernet_key"`)

**Impact:** If v2's `keyring` crate is called with the spec's names, it will find no existing credential in Windows Credential Manager. It will then generate a brand-new Fernet key and store it under the wrong name. All v1-encrypted data (TOTP secrets, Agent Zero API key, SMTP password) will fail to decrypt with `InvalidToken`. The user's existing TOTP enrollment and Agent Zero integration become inoperable. Recovery requires re-enrolling MFA and re-entering the Agent Zero API key.

**Additional note:** The v2 spec also conflates two encrypted values. The Fernet key protects *both* the TOTP secret (via `auth.py`'s `crypto.encrypt()`) and the Agent Zero API key (stored as `agent_zero_api_key_encrypted` in `config.json`) and the SMTP password (`smtp_password_encrypted` in `config.json`). The spec §7 only mentions TOTP secret and Agent Zero API key — confirm the SMTP password is also on the migration radar.

**File fallback:** v1's `crypto.py` also has a keyfile fallback at `%APPDATA%\DFARS\.keyfile`. If the current install used this fallback (e.g., keyring was unavailable at first run), v2 must also read and honor this path before attempting key generation. The Rust implementation must replicate the same fallback priority: keyring first, then `.keyfile`, then generate new.

**Verdict:** BLOCKING. This is MUST-DO item #1.

### 2.7 Single-User Enforcement

**v1 behavior:** `create_user()` checks `user_exists()` before inserting and raises `AuthError` if a user already exists. There is also a `UNIQUE` constraint on `username` in the schema, which provides a belt-and-suspenders DB-level guard.

**v2 spec:** Spec §2 and §7 both state single-user enforcement is preserved. The `auth_setup_first_run` command corresponds to `create_user`.

**Analysis:** The check-then-insert pattern in v1 has a theoretical TOCTOU race (two parallel calls to `auth_setup_first_run` could both pass the `user_exists()` check, then one INSERT fails on the UNIQUE constraint and returns an error). In the Tauri single-process context, Tauri commands are async but share the same `AppState` — using a `Mutex<bool>` `setup_in_progress` flag would close the race, but this is a negligible risk for a desktop app with one user. The DB-level UNIQUE constraint is the reliable backstop.

**Verdict:** PASS. Preserve the application-layer check and rely on the UNIQUE constraint as the authoritative guard. Document that the UNIQUE constraint is the actual enforcement mechanism.

### 2.8 API Tokens — Bearer Token Design

**v1 behavior:** Token format `"dfars_" + secrets.token_urlsafe(32)` (~256 bits of entropy). Stored as Argon2id hash + 12-char preview in `api_tokens` table. Verification in `api_tokens.verify()` fetches all tokens and tries each hash sequentially. With `~100ms` per Argon2 verify and a realistic N of 1–3 tokens, latency is acceptable. The `dfars_` prefix enables secret-scanning tools to flag leaks.

**Weaknesses identified:**

1. **Linear scan with no fast-path identifier.** The code itself notes this limitation. For N=1-3 tokens this is fine. If N grows (e.g., 20+ tokens for CI automation), cumulative Argon2 cost (~2 seconds) becomes an external API latency issue. The fix is a fast-path: store an HMAC-SHA256 fingerprint of the token alongside the Argon2 hash, compute the fingerprint from the incoming plaintext in ~microseconds, use it to narrow to one candidate row, then verify with Argon2. The fingerprint is not a secret (it derives from the plaintext token which the holder already knows) but it is not reversible from the stored fingerprint alone.

2. **No token expiry.** Tokens are permanent until explicitly revoked. For Agent Zero integration this is acceptable (the token is machine-to-machine). Worth noting for future multi-user expansion.

3. **`agent_zero_auto_setup` hardcodes a filesystem path.** `auth_routes.py` line 379: `plugin_dir = Path("C:/Users/jhenn/agent-zero/agent-zero/usr/plugins/_dfars_integration")`. This is a development artifact — the path contains the developer's username and will fail on any other machine. This route is in the v1 codebase and will be ported to v2. It must not be ported as-is.

**Verdict:** PASS with SHOULD-DOs. The core token design is sound. The hardcoded path is a SHOULD-DO for the v2 port.

### 2.9 Chain-of-Custody Legal Integrity — Auth Guard Coverage

**v1 behavior:** `flask_app.py`'s `before_request` hook (`_require_auth`) enforces session authentication for all routes except `static`, `auth.*`, `dfars.health`, and `api_v1.*` (the last uses bearer token auth in `api_routes.py`). The custody mutation routes (`/case/<id>/custody/add`, etc.) fall under `dfars.*` and are thus session-gated. The API equivalents are bearer-token gated.

**v2 concern:** In v2, the auth gate moves from a Flask before_request hook to individual Tauri command implementations. There is no equivalent framework-level interceptor unless the team builds one. Each command in `custody_add`, `custody_update`, `custody_delete` must individually call a session validation function before executing the DB mutation. If any custody command is added without this call — even temporarily during development — it is an unauthenticated write path to legally significant data.

**Recommended pattern:** Implement a `require_session(state: &AppState, token: &str) -> Result<SessionData, AppError>` helper in `auth/session.rs`. Every Tauri command that touches `chain_of_custody`, `evidence`, `hash_verification`, or `tool_usage` must call this helper as the first line of the command body. The QA gate for Phase 1 must include a test that verifies calling any auth-required command without a valid session token returns `AppError::Unauthorized`.

**Verdict:** FLAG — not a v1 bug, but a design requirement for v2 that must be explicitly enforced in every custody command. This is MUST-DO item #3.

## 3. Required Changes Before Implementation

These are hard gates. Phase 1 code MUST NOT be written until items 1 and 2 are confirmed resolved. Item 3 is a design requirement to be enforced throughout Phase 1 and subsequent phases.

**MUST-DO 1 — Fix keyring service/account names in v2 spec and implementation.**
The v2 Rust `keyring` crate call for reading the Fernet key must use:
- `service = "DFARS Desktop"` (exact string, space included, case-sensitive)
- `username/account = "totp_encryption_key"` (exact string)

Update the v2 spec §7 to reflect the correct names. The implementation in `crypto.rs` must also replicate the file fallback: if the keyring returns no result, read `%APPDATA%\DFARS\.keyfile` before generating a new key. Add a startup-time diagnostic log at INFO level that records which key source was used (keyring, keyfile, or new) without logging the key value.

**MUST-DO 2 — Monotonic lockout timer in v2.**
Do not replicate v1's `datetime.now()` wall-clock lockout in Rust. Implement as follows:
- Add `failed_attempts: Mutex<HashMap<String, (u32, Option<std::time::Instant>)>>` to `AppState` (or a dedicated `AuthState` struct). Keys are usernames; values are `(count, lockout_expires)`.
- On process startup, hydrate this map from the DB: if a user row has `failed_login_count >= MAX_FAILED_ATTEMPTS` and a non-null `locked_until`, set `lockout_expires = Some(Instant::now() + remaining_duration)` where `remaining_duration` is derived from `locked_until` compared to `SystemTime::now()`. If the DB shows the lockout has already expired at startup, clear it.
- All lockout checks and updates during the process lifetime use `Instant`, not wall-clock. Persist `failed_login_count` and `locked_until` to SQLite only for cross-restart durability, not for runtime decisions.
- The `locked_until` column in SQLite remains as written today (ISO string of wall-clock time at lock creation) — this is only used at process startup to initialize the in-memory `Instant`.

**MUST-DO 3 — Mandatory session guard on all custody/evidence mutation commands.**
Implement `require_session(state: &AppState, token: &str) -> Result<SessionData, AppError>` in `auth/session.rs`. Document in the codebase (a comment block in `commands/mod.rs`) that this function MUST be the first call in any command that mutates `chain_of_custody`, `evidence`, `hash_verification`, `tool_usage`, `analysis_notes`, `entities`, `entity_links`, or `case_events`. The QA engineer must add a negative test for each of these command groups verifying `AppError::Unauthorized` on missing/invalid session.

## 4. Recommended Improvements

These are SHOULD-DOs: worth doing if they fit within Phase 1 scope and don't extend the timeline materially.

**SHOULD-DO 1 — Explicit Argon2 parameters for new hash generation.**
When v2 generates new hashes (initial user creation, password change, recovery code hashing), use `Params::new(65536, 3, 4, None)` rather than `Argon2::default()`. This maintains equivalent work-factor to v1 for all new credentials, avoids a silent security downgrade (the Rust default is weaker), and ensures parity across v1/v2 installs sharing the same DB.

**SHOULD-DO 2 — Rate limit recovery code verification attempts.**
v1 has no rate limit on recovery code attempts at the MFA verify step. v2 should apply a separate failed-attempt counter for MFA (TOTP + recovery code failures combined). After N failures at the MFA step (suggest 5), clear `mfa_pending` from the session and force the user back to the password step. Log the MFA failure. This prevents an in-session attacker from exhausting recovery codes at Argon2 speed (~10 attempts/second would take ~100 seconds to exhaust 10 codes — too fast for a forensic app).

**SHOULD-DO 3 — TOTP brute-force protection.**
v1 has no per-attempt lockout at the MFA verify step — only at the password step. An attacker who passes password auth (or has a valid session token) can try TOTP codes indefinitely. The TOTP keyspace is 000000–999999 (1,000,000 values). With a 30-second step and 2 valid windows, an attacker gets at most 3 valid codes per 30 seconds, so exhaustion is time-bounded by the TOTP clock, not by the app. However, implement a counter in `AppState` that tracks consecutive TOTP failures per pending session token. After 10 consecutive TOTP failures, invalidate the pending session entirely (forcing re-authentication from password step) and log the event.

**SHOULD-DO 4 — Add fast-path fingerprint to API token verification.**
In `tokens.rs`, store an HMAC-SHA256(key=`token_hash`, msg=`plaintext[..8]`) fingerprint alongside the Argon2 hash. On verification, compute the fingerprint of the incoming token and use it to narrow to a single candidate row before running Argon2. This makes token verification O(1) Argon2 operations regardless of token count. The HMAC key is the stored `token_hash` (already a secret) and the message is the first 8 chars of the token — enough for a unique fingerprint, not enough for preimage. Note: a simpler approach is to use the first N chars of the raw token as the lookup key (the `token_preview` field already does this to 12 chars). Using `token_preview` directly as the lookup key would reduce the Argon2 calls to 1, at the cost of revealing that the incoming token matches `token_preview` (which is already displayed in the UI). This is acceptable and is the simpler path.

**SHOULD-DO 5 — Remove hardcoded filesystem path from Agent Zero auto-setup.**
`auth_routes.py` line 379 hardcodes `C:/Users/jhenn/agent-zero/...`. When this route is ported to v2 (`settings_set_agent_zero` or a dedicated auto-setup command), the plugin path must be configurable (read from `config.json` or derived from the running Agent Zero container's volume mount). Default should use a path relative to a configurable Agent Zero root, not an absolute developer path. This is a v1 bug that must not be ported.

**SHOULD-DO 6 — Document the keyfile fallback security boundary.**
The `%APPDATA%\DFARS\.keyfile` fallback stores the Fernet key as plaintext adjacent to `auth.db`. If keyring (Windows Credential Manager) is available, this fallback should never be used on Windows — and v2 should warn visibly in the UI if startup detects that the keyfile fallback is in use. Add a `settings_get_security_posture()` command that returns `{ keyring_active: bool, mfa_enabled: bool, recovery_codes_remaining: u32 }` so the UI can surface a "Upgrade to keyring" prompt if the file fallback is active. This is pre-existing in v1 as a silent log warning; v2 should elevate it.

## 5. Open Questions

**OQ-1 — Keyring credential for Agent Zero API key: same entry or separate?**
v1 stores the TOTP Fernet key under `service="DFARS Desktop", account="totp_encryption_key"`. The Fernet key is then used to encrypt both the TOTP secret *and* the Agent Zero API key and SMTP password (all stored as `*_encrypted` fields in `config.json`). Is this intentional — one key protects all secrets — or was separate per-secret keyring entries considered? This affects the v2 `crypto.rs` design (one keyring read vs. multiple). The current single-key design is fine; this question is only about confirming it is intentional.

**OQ-2 — Session token transport in Tauri: IPC return value vs. cookie?**
The spec §7 says "an HMAC cookie issued to the WebView on login." Does this mean a literal HTTP cookie set via Tauri's WebView API, or a token returned from the `auth_login` command and stored in React state? Recommend the latter — React state or `sessionStorage` — to avoid any ambiguity about cookie lifetime, `HttpOnly`, and JS-accessible cookie semantics inside the WebView. This must be decided before Phase 1 frontend work begins.

**OQ-3 — Session lifetime: restart-invalidation or persistence?**
v1's session survives app restarts (7-day lifetime from the persisted session cookie). v2's in-memory map inherently loses all sessions on restart. Is this acceptable? For a forensics app, requiring re-authentication after restart is a security improvement (NIST 800-63B §7.2 recommends reauthentication after a period of inactivity or restart). Confirm with the user that they accept this behavior change.

**OQ-4 — SMTP password Fernet migration scope.**
The v2 spec §7 mentions TOTP secret and Agent Zero API key as Fernet-encrypted values that v2 must read. The SMTP password (`smtp_password_encrypted` in `config.json`) is also Fernet-encrypted via `app.crypto`. Confirm this is in scope for the v2 crypto module from day one, not deferred to Phase 5 (when SMTP is added). If v2 launches without SMTP password decryption and the user has SMTP configured, the first v2 launch will silently fail to send emails.

## 6. Sign-Off Conditions for SEC-1 Final Approval

The following must be true in the Phase 1 implementation PR before SEC-1 is closed:

1. `crypto.rs` reads the Credential Manager entry with service `"DFARS Desktop"` and account `"totp_encryption_key"`, with `.keyfile` fallback at `%APPDATA%\DFARS\.keyfile`. A test with a known credential confirms the correct entry names.

2. `auth/session.rs` uses `std::time::Instant` for lockout expiry (not `SystemTime`, not wall-clock). The test suite includes a scenario that simulates 5 failed attempts, advances a mock clock, and confirms lockout is enforced independent of system time.

3. `require_session()` exists in `auth/session.rs` and is called as the first statement in every command that touches `chain_of_custody`, `evidence`, `hash_verification`, `tool_usage`, `analysis_notes`, `entities`, `entity_links`, and `case_events`. A negative test confirms `AppError::Unauthorized` for each command group when no valid session is provided.

4. A Rust unit test generates an Argon2id hash with `Params::new(65536, 3, 4, None)` and verifies it successfully. A second test takes a known v1-generated hash string (captured from the real auth.db or generated in Python with the same params) and verifies it with the Rust implementation — confirming cross-library compat.

5. `totp-rs` is initialized with `step=30`, `digits=6`, `skew=1`. A test generates a TOTP code for a known secret and timestamp in Rust and compares against the expected value from RFC 6238 test vectors or a known pyotp output.

6. Open questions OQ-2 (session token transport) and OQ-3 (session lifetime on restart) are answered and the answers are reflected in the spec update before Phase 1 frontend work begins.

7. The Phase 1 PR does not contain any reference to keyring service `"dfars_desktop"` or account `"fernet_key"`.

## Appendix: v1 Behaviors Confirmed as PASS (No Action Required)

- **Constant-time username enumeration guard:** `_DUMMY_HASH` pattern in `verify_password` is sound. v2 must replicate with a pre-hashed dummy value loaded at startup.
- **Password input length cap (1024 chars):** Correct. Prevents memory DoS from huge Argon2 inputs. Port unchanged.
- **Username character allowlist `[a-zA-Z0-9._-]`:** Correct. Port unchanged.
- **Single-use recovery code marking (`used_at` timestamp):** Schema enforces this. Port unchanged.
- **`disable_mfa` requiring password re-entry:** Correct defense against session hijack. Port unchanged.
- **Recovery codes revoked on MFA disable and re-enrollment:** Port unchanged.
- **API token `dfars_` prefix for secret scanning:** Port unchanged.
